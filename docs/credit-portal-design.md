# 고객 창구 포털 설계 문서 (Credit / Renewal Portal)

> 상태: **설계 확정(크레딧 배분 자동화 제외)** · 작성 기준일 2026-06-17
> 본 문서는 기존 Exocad Serial Manager 프로젝트에 **추가될 고객 대면 웹 포털**의 설계서입니다.
> 코드 착수 전 검토용이며, 확정된 결정사항을 근거로 작성되었습니다.

---

## 1. 개요 / 범위

- **목적**: 이미 제품을 보유한 고객이 **AI 크레딧 신청 / 갱신 중단 / 갱신 재개**를 직접 접수하는 한정 공개 웹 포털.
- **역할**: 포털은 **접수 창구**만 수행. 실제 DB 쓰기·메일·Playwright 자동화는 **메인 프로젝트가 처리**.
- **대상**: 기구매 고객 한정(시리얼 보유자). 일반 공개 아님.
- **현재 상태(코드 확인됨)**:
  - 크레딧 관련 로직/데이터/UI **전무** → 전부 신규.
  - `my.exocad ID`는 현재 데이터 모델에 **없음**. 존재하는 건 단일 관리자 SSO 계정 `exocad_username`뿐.
  - 고객 시리얼은 **항상 8-4-8(20자) 표준 형식** 유지(비표준 없음).

---

## 1.1 시스템 맥락 / 경계 / 설계 원칙

**역할 — 중간 teller**: 본 프로젝트·포털은 판매사와 고객 사이의 **접수·중계(teller)** 역할이다.

- **판매사 사이트 `partner.exocad.com`**: 실제 상품 판매·관리·배포 주체. 주문/갱신중단/크레딧 배분 등 외부 작업은 **Playwright로 이 사이트에 접속·조작**한다. (취소 자동화가 쓰는 단일 관리자 SSO 계정과 동일 계열)
- **ERP(외주·변경권한 없음)**: 판매 완료 후 고객 청구서 발송·정산은 외주 ERP에서 처리. **API 연계·자동 연동 불가** → 본 시스템은 ERP와 **연계하지 않는다.** 청구/정산은 사람이 ERP에서 수동 처리하며 **본 시스템 범위 밖**이다.

**설계 원칙 (반드시 준수)**:

- **단순성 우선**, 한정된 자원(무료 VM 등) 내에서 작동.
- **복잡도 증가로 인한 fatal case 경계** → 데이터·권한이 불충분한 자동화는 **하지 않고**, **메일/관리자 수동 핸드오프**로 안전하게 처리한다.
- 위험하거나 미완성인 자동 동작은 **기본 OFF + 설정 토글**로 두고, 데이터·검증이 준비됐을 때만 켠다.

## 2. 확정된 결정사항 (제약)

| 항목 | 결정 |
|---|---|
| 공개 범위 | 기구매 고객 한정 |
| 비밀번호 | DB에 **bcrypt 해시 저장**, **재설정 전용**(찾기/조회 없음) |
| 고객 프로필 정보 | 비밀번호 제외 전 필드 **관리자 열람·수정 가능**(admin pw 뒤). DB 변경 시 포털에 **지연 동기화** |
| 로그인 식별자 | **별도 로그인 ID**(가입 시 생성). my.exocad ID와 무관 |
| my.exocad ID | 로그인과 무관한 **프로필성 정보**. 크레딧 신청 시 입력/선택 |
| 시리얼 본인확인 | 시리얼 보유 = 소유 증명으로 사용 |
| 가입 이메일 검증 | **생략**(한정 공개) |
| 제품 보유 노출 | 허용(민감정보로 보지 않음) |
| 데이터 접근 | 포털은 **인증된 메인 API만 호출**, 직접 DB 접근 금지 |
| 크레딧 배분 자동화 | **후속 설계**(exocad 크레딧 화면 확보 후) |
| 갱신 재개 | **접수만**, 관리자 수동 처리 |
| 갱신 중단 | 접수 → 관리자 승인 후 자동 / **만료 윈도우 미처리 시 failsafe 자동** |
| 크레딧 자동배분 | **현재 불가(데이터 부족)**. 후속 **모듈** + **설정 토글(기본 OFF)**. 켜지면 자동 배분 |
| 크레딧 신청 처리 | 토글 OFF 동안: 신청을 **지정 메일로 발송** → 관리자 수동 |
| ERP 연계 | **없음**(외주·변경권한 없음). 청구/정산은 ERP에서 수동, 범위 밖 |
| 외부 자동화 대상 | `partner.exocad.com` (Playwright) |

## 3. 아키텍처

```
[고객 브라우저]
   │  HTTPS (Cloudflare Tunnel)
   ▼
[포털 프론트엔드]  ── 인증된 포털 API ──▶  [메인 프로젝트(서버 모드)]
                                              ├─ SQLite (단일 VM)
                                              ├─ 메일(SMTP 템플릿)
                                              └─ Playwright(취소/크레딧 배분)
```

- **호스팅**: GCP 무료 e2-micro VM 1대. 메인 프로젝트(서버 모드) + 포털이 **동일 백엔드·동일 DB** 공유.
- **노출**: **Cloudflare Tunnel**(HTTPS 자동, 포트 미개방, VM IP 은닉). duckdns/수동 인증서 불필요.
- **DB**: SQLite 유지(이 규모에 충분). **`.db` 일일 백업(GCS 등) 필수**.
- **Playwright 안정화**: 하루 ≤5회. **swap 2GB** + 런처에 **`--disable-dev-shm-usage`** 추가(저메모리 크래시 방지). 브라우저 1개 직렬 실행(기존 큐 재사용).
- **관리자**: 기존 admin 인증으로 전체 고객정보 열람·수정.

## 5. 보안 설계 (전부 적용)

- 비밀번호: **bcrypt 해시**, **재설정 전용**(등록 이메일로 일회용·단기만료 링크 → 새 비번 설정 → 최초 로그인 강제 변경)
- 세션 쿠키 `HttpOnly` + `Secure` + `SameSite`, 상태변경 POST에 **CSRF 토큰**
- **레이트리밋 + 시도횟수 제한**, 에러 메시지 **일반화**("정보가 일치하지 않습니다"로 통일 — 열거 방지)
- 비밀번호 규칙(최소 8자, 대/소문자+숫자) **서버에서도 강제**. UI에 규칙 표시
- 설정/입력 문구를 웹·메일에 렌더링 시 **XSS 이스케이프**
- 모든 신청은 서버가 **"로그인 계정 소유"인지 재검증**(클라이언트가 보낸 serial/id 신뢰 금지)
- 시리얼 **마스킹 표시**(`2RD5XXXX-XXXX-XXXX98R2`), 모듈 등 기타정보 미표시
- HTTPS 강제(Cloudflare Tunnel)
- 관리자 열람 시 **기존 해시 admin 자격증명** 재사용(평문 신설 금지)
- 데이터 매치 실패 시 **PM@geomedi.co.jp 안내 팝업**

## 6. 기능 사양

### 6.1 가입 / 로그인
- 가입 입력: 고객명, 이메일, 전화, 실제주소, my.exocad ID, 로그인 ID, 비밀번호, 비밀번호 확인.
- **로그인 ID 중복검사**(고유 보장). 비밀번호 규칙 표시·서버 검증.
- 가입 직후 자동 로그인 → 초기설정으로 이동.
- 로그인 페이지: **비밀번호 재설정**(찾기 없음).
- **언어 선택**(로그인/가입 페이지에서) → 계정에 영속.

### 6.2 비밀번호 재설정
- 입력: 로그인 ID + 이메일. 매치 시 **등록 이메일로 일회용·단기만료 링크** 발송 → 새 비번 설정.
- 메시지 일반화(열거 방지), 레이트리밋.

### 6.3 초기설정 (계정 ↔ 고객 연결)
- 가입정보 중 DB와 매치되는 항목 있으면 제품명 표시 + 시리얼 입력 요청.
- 시리얼 입력 UI: **8 / 4 / 8 박스**(숫자·영문), 박스 채워지면 **자동 다음 박스 이동**.
- 매칭: `serialService.getBySerialNumber()`(대소문자 무시). 매치 시 해당 `customer_id`를 `portal_account_links`에 연결.
- 다수 시리얼 보유 시 **하나라도 매치되면 성공**.
- **다중 제품/ID 취합**: 로그인 시 + 주기적 **지연 매칭**으로 고객 데이터 변경분을 포털에 반영(빈번 매칭 없음, 부하 최소).
- 매치 실패 시 안내 팝업(PM 메일 연락).

### 6.4 유저 정보 페이지
- 우상단 유저 아이콘 → 진입.
- 연결된 고객의 **제품명(전체)** + **시리얼(마스킹)** 표시. 모듈 등 기타 미표시.
- **비밀번호 변경**(규칙 표시·서버 검증).

### 6.5 AI 크레딧 신청
- 입력: **my.exocad ID**(프로필에서 선택 또는 입력) + **패키지 선택**(설정의 크레딧 수량·금액).
- 안내: 발급 후 **취소/환불 불가, 유효기간 2년**.
- 제출 → `portal_requests(type=credit)` 적재 **+ 지정 메일 주소로 신청 내용 발송**(관리자 수동 처리).
- **자동 배분 토글(설정, 기본 OFF)**:
  - **OFF(현재 기본)**: 위처럼 **메일 핸드오프만** — 관리자가 partner.exocad.com에서 수동 배분.
  - **ON(후속 모듈 준비 시)**: 관리자 승인 후 Playwright 자동 배분(§8).
- 완료 팝업 + 신청 확인 메일.
- ⚠️ 실제 배분 자동화 = **데이터 부족으로 현재 불가** → 충분해지면 **모듈로 추가**(§8).

### 6.6 갱신 중단 요청
- 입력: **본인 소유 시리얼**(서버가 소유 검증, 단순 "DB 매치" 아님).
- 안내: 중단 후 90일 내 업데이트 비용만으로 재개 가능, 이후 리뉴 금액 추가(문구 설정 가능).
- 처리 규칙:
  - **평소**: `portal_requests(type=renewal_stop)` → 관리자 승인 → `serialService.setStopRequested(true)` → 기존 자동취소 흐름.
  - **failsafe(자동, 관리자 승인 없이)**: 대상 시리얼 만료가 **윈도우(오늘/내일)** 인데 미처리(휴먼에러)거나 당일 접수라 검토 시간이 없으면, 기존 `runCandidateFailsafeCancelNow` 크론이 자동 확정·취소.
  - 통합 포인트: failsafe 스캔이 **inbound 메일뿐 아니라 포털 발 중단요청도 포함**하도록 확장 필요.
- 완료 팝업 + 확인 메일.

### 6.7 갱신 재개 요청
- 현재 보유 시리얼 목록 + 체크박스 표시(서버가 본인 소유 재검증).
- 체크 시 **신청 버튼 + 견적서 신청 버튼** 활성화.
- **접수만, 관리자 수동 처리** (자동 플래그 해제/재갱신 호출 ❌).
- 신청 버튼: 금액 확인 필요 시 "견적서 버튼을 눌러달라" 팝업 후 신청 재노출(문구 설정).
- 견적서 버튼: "영업일 3일 내 이메일로 견적서 전달 예정" 안내(문구 설정).
- 완료 팝업 + 확인 메일.

### 6.8 창구 설정 (관리자, 메인 프로젝트 설정 내)
- **크레딧 자동배분 ON/OFF 토글**(기본 OFF) + **크레딧 신청 수신 지정 메일 주소**.
- 크레딧 패키지(수량·금액), 각 안내 문구(중단/재개/견적), 언어 등 — `settings`(key-value JSON) 재사용.
- 전 고객 포털 계정·신청 **열람·수정**(admin pw 요구).

### 6.9 i18n / 언어
- 포털 UI 문자열: i18n(ko/en/ja). 하드코딩 금지.
- **안내 문구(설정값)는 i18n과 별개로 언어별 저장**.
- 언어는 로그인/가입 페이지에서 설정 → 계정에 영속(비로그인 단계는 브라우저/쿠키 기반).

## 7. 엣지케이스 처리 요약

| 케이스 | 처리 |
|---|---|
| 로그인 ID 중복 | 가입 시 차단 |
| 이메일 오타(미검증) | 재설정 불가 → PM 메일 안내 |
| 동일 이메일 다계정 | 재설정 대상 식별 규칙 필요(로그인 ID 기준) |
| 다중 customer 레코드(제품 일부 누락) | 지연 매칭으로 취합. 추가 시리얼 인증도 허용 |
| 비표준 시리얼 | 표준만 존재(확정). 혹시 섞인 `IMPORT-`류는 포털에서 무시 |
| 타인 시리얼/한 시리얼 다계정 | 시리얼=소유증명(수용). 중복계정 정책은 운영 결정 |
| cancelled/expired 시리얼에 중단요청 | 무의미 → 막거나 안내 |
| 중단요청 멱등 | `setStopRequested` 동일값 no-op(OK) |
| 클라이언트가 보낸 serial/id 위조 | 서버가 본인 소유 재검증 |

## 11. 구현 에이전트 인수인계 지침 (반드시 준수)

> 구현은 본 설계 대화 맥락이 **없는 상태**에서 시작됩니다. 아래를 먼저 읽고 그대로 따르세요.

### 11.1 시작 전 필독
- 이 문서 **전체** + 저장소 루트 **`CLAUDE.md`**(프로젝트 규칙) + **`AGENTS.md`** 를 먼저 읽을 것.
- 작업 브랜치: **`feature/credit-system`**(이 문서가 커밋된 브랜치). `main`은 안정 baseline이므로 **직접 수정 금지**.
- 비기술 PM과 협업: **Phase 단위로 진행하고 각 Phase 완료 시 확인받을 것.** 전체를 한 번에 구현하지 말 것.

### 11.2 아키텍처 하드 규칙 (CLAUDE.md 발췌 — 위반 시 빌드 깨짐)
- IPC 추가 4단계: ① `src/shared/types.ts`의 `IPC_CHANNELS`에 채널 추가 → ② `src/main/ipc-handlers.ts`에 핸들러 → ③ `src/main/preload.ts`에 노출(**채널 문자열 인라인**) → ④ 렌더러에서 `window.electronAPI` 호출.
- **`preload.ts`는 외부 import 금지**(샌드박스에서 `require` 차단). 채널 문자열을 `shared/types`에서 import하지 말 것.
- 비즈니스 로직은 `src/main/services/`, 공유 타입은 `src/shared/types.ts`.
- 타입체크: `tsc --noEmit`(렌더러) / `tsc --noEmit -p tsconfig.main.json`(메인). 테스트: `npx vitest run`.
- 네이티브 의존성 변경 시 `npx electron-rebuild -f -w better-sqlite3`.
- 포털은 서버 측(`src/server/`)/API 기반으로 구현. 포털에서 **DB 직접 접근 금지 → 인증된 API만 호출**.

### 11.3 절대 하지 말 것 (DO NOT)
- ❌ **크레딧 자동 배분(Playwright) 구현 금지** — `partner.exocad.com` 크레딧 화면 셀렉터/흐름이 아직 없음(§8). 정보가 제공되기 전엔 손대지 말 것. 지금은 **크레딧 신청 → 지정 메일 발송 + DB 적재**까지만.
- ❌ **피처 플래그(`portal_enabled`, 크레딧 자동배분 토글)는 기본 OFF 유지.** 임의 ON 금지.
- ❌ **ERP 연계/외부 API 호출 금지**(외주·권한 없음, 범위 밖).
- ❌ **baseline 동작 변경 금지** — 기존 취소/메일/폴링 기능을 바꾸지 말 것. 특히 `cancel.service.ts`의 **취소 대상 검증 가드 / `verified` 게이트를 약화·제거하지 말 것**.
- ❌ **비밀번호 평문·env 저장 금지** → DB에 bcrypt 해시. **비밀번호 조회/표시(찾기) 기능 만들지 말 것** → 재설정만.
- ❌ **클라이언트가 보낸 serial/id 신뢰 금지** → 모든 신청은 서버에서 "로그인 계정 소유" 재검증.
- ❌ 여러 Phase를 한 번에 몰아 구현하지 말 것.

### 11.4 반드시 재사용 (신규 작성 금지)
